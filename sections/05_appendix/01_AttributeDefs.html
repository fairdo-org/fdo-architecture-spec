List of attribute definitions used in this document

The lists of attributes given in C.1.1 - C.1.3 are very early drafts and currently under discussion. The discussion space can be found here.
To achieve definition of high-quality attributes, each attribute definition has to settle on:
  • the attribute name
  • the attribute description
  • the attribute cardinality
  • the attribute validation mechanism (syntax, union, inlineCombination, attribute referencing, profile referencing, external vocabulary referencing)
  • Depending on the validation mechanism an attribute uses, the following needs to be specified:
    • syntax: a syntax definition belonging to this attribute
    • union: several attribute definitions comprising this union
    • inlineCombination: several attribute definitions comprising this inline combination, and potentially a cardinality to override the original attribute cardinality
    • attribute referencing: a list of attributes which can be pointed at
    • profile referencing: a list of profiles which can be pointed at
    • external vocabulary referencing: always points at 0.FDO/ImportsVocabulary
Discussion workflow: When the group has agreed an a specific cell in one of the tables, one editor is chosen to make the changes in the table. We use the following color code to guide the discussion:
  1. red: discussion did not start yet, or currently under discussion
  2. green: discussion finalized
Colors will be removed after discussions have concluded.

Mandatory attributes

This list of attributes is a draft of the mandatory attributes as defined by the FDO Forum.
Table 2: Attribute definitions that MUST be used in any FDO.
PID Human-readable name Description Cardinality Validation mechanism Validation mechanism specific information
0.FDO/Type FDO_Type This attribute specifies the type of the resource being represented by the FDO. The purpose of the FDO type is to give a hint to a machine how this object may be processed. Each FDO must have at least one FDO type but can also have multiple FDO types. The FDO type typically points to one of the following attributes that specify the FDO type more closely: ... (tbd). But other values of the attributes are possible as well. This attribute is defined by the FDO Forum and is part of the list of mandatory FDO attributes. 1..* Attribute referencing Reference to any string allowed (no pre-defined list of allowed data types/values)
0.FDO/Profile FDO_Profile This attribute specifies the profile used by the FDO. The purpose of the FDO profile is specify which attributes must/may be present in the FDO, and which cardinality they have in the given context. Each FDO must have at least one FDO profile but can also have multiple FDO profiles. The FDO profile references another FDO that defines a profile. The reference can be of any syntax. This attribute is defined by the FDO Forum and is part of the list of mandatory FDO attributes. 1..* Profile referencing Reference to a profile FDO (0.FDO/ProfileDef)
0.FDO/Data FDO_Data This attribute indicates a path to the data (also called: bit-sequence) of the FDO. The purpose of the FDO data reference is to signal a machine where it can find the data associated to this FDO. Each FDO must have at least one FDO data reference but can also have multiple FDO data references. The FDO data reference typically points to one of the following attributes that specify the FDO data reference more closely: ... (tbd). But other values of the attributes are possible as well. This attribute is defined by the FDO Forum and is part of the list of mandatory FDO attributes. 1..* Attribute referencing Reference to any string allowed (no pre-defined list of allowed data types/values)
